This project involved designing and provisioning a containerized platform infrastructure from the ground up using infrastructure-as-code, with security considerations embedded at every layer — from how the cluster is shaped to how identities are granted access within it.

The repository is private. This writeup covers the architecture decisions, security model, and automation approach without exposing implementation details.

The Problem Worth Solving

Spinning up a Kubernetes cluster manually and granting broad IAM permissions to get things running quickly is a pattern that causes real problems at scale. Over-provisioned identities accumulate. State drifts. Nobody remembers what was clicked in the console six months ago.

This project was an exercise in doing the opposite — building a platform where every resource exists because code says it should, every identity has exactly the access it needs, and the only way to change the infrastructure is through a version-controlled pipeline.

Infrastructure Design

The platform is built on Amazon EKS and provisioned entirely through modular Terraform configurations. The Terraform layout separates concerns deliberately — networking, cluster, and access control are not tangled together. Each module has a clearly bounded responsibility, which makes changes easier to reason about and review.

The cluster spans multiple availability zones by design, not as an afterthought. Node groups are configured declaratively, meaning capacity decisions are documented in code and reproducible on any environment rather than locked inside a one-off console session.

IAM Architecture

Access control was the most deliberate part of this build. IAM roles were scoped narrowly at each boundary in the stack — the role the cluster uses to manage AWS resources, the role that node groups assume when pulling images and registering with the control plane, and the role that the pipeline uses to drive provisioning are all distinct, each carrying only the permissions their function requires.

The goal was to ensure that no single credential compromise could propagate beyond its intended scope. A misconfigured node should not be able to make cluster-level API calls. A pipeline runner should not be able to do anything outside its provisioning scope. Each identity is a narrow instrument.

Pipeline Automation

A GitHub Actions workflow drives the full infrastructure lifecycle. Provisioning, validation, and teardown are all triggered through version-controlled events — there is no separate runbook, no console step hidden between pipeline stages.

The pipeline authenticates into AWS using scoped credentials and executes Terraform operations in sequence. State is managed consistently across runs. The result is an infrastructure lifecycle that is auditable, repeatable, and tied to the git history of the project.

Design Decisions

Why Terraform over console provisioning?

Console-provisioned infrastructure creates invisible state. There is no record of why a security group rule exists or what depended on it. Terraform forces every resource to be declared, every dependency to be explicit, and every change to go through a plan before it applies. The infrastructure becomes something you can read, review, and reason about — not something you have to reverse-engineer.

Why modular configurations rather than a flat layout?

A single flat Terraform file works fine until it doesn't. Modules enforce a boundary between what something does and how it does it. They make it possible to change the networking layer without touching the cluster configuration, or to reuse node group patterns across environments without duplicating logic. The upfront cost is structure. The payoff is maintainability.

Why scope IAM roles per boundary rather than sharing them?

A shared role with broad permissions is a convenience that hides risk. When a node group and a pipeline runner share credentials, the compromise surface of each expands to include everything the other can do. Narrow roles make the blast radius of any failure predictable. They also make the intent of each identity legible in the policy.

Why pipeline-only lifecycle management?

Manual infrastructure changes introduce drift — the gap between what the code describes and what actually exists in the environment. Once that gap exists, the codebase is no longer the source of truth. Restricting all changes to the pipeline eliminates that gap by design. It also means every infrastructure change is reviewed, logged, and reversible.

Outcome

The platform was successfully provisioned, validated, and torn down exclusively through the pipeline — no manual console intervention at any stage of the lifecycle. IAM roles were verified to hold minimum required permissions at each boundary, and no over-provisioned access was present in the final configuration.

This project represents a shift from infrastructure that is built to one that is declared — where security is a structural property of the system rather than a layer applied after the fact.
