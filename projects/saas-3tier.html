High Availability 3-Tier SaaS Architecture on AWS

Architecture 3-Tier High Availability
Services VPC · ALB · EC2 · ASG · RDS
Availability Multi-AZ
Networking Public + Private Subnets

This project demonstrates a production-style web application architecture designed for high availability and resilience. The system distributes traffic across multiple availability zones using an Application Load Balancer and Auto Scaling Group while automatically replacing failed instances.

Objective

  • Design a resilient multi-AZ web infrastructure
  • Distribute traffic using an Application Load Balancer
  • Automatically replace unhealthy compute instances
  • Implement horizontal scaling during load spikes

Architecture Overview

User → Internet-Facing ALB → Target Group → Auto Scaling Group → EC2 Instances

Incoming traffic is distributed across EC2 instances running in multiple availability zones. Health checks ensure traffic is routed only to healthy instances, while the Auto Scaling Group automatically launches replacement instances if failures occur.

Application Load Balancer
  • Type: Application Load Balancer (internet-facing)
  • IP address type: IPv4
  • Availability Zones: 2
  • Routes requests to EC2 instances through target groups
Target Group Configuration
  • Protocol: HTTP · Port: 80 · Target type: Instance
  • Health check path: "/" · Healthy threshold: 2
  • Unhealthy instances automatically removed from routing pool
Auto Scaling Group
  • Minimum: 1 instance · Maximum: 2 instances · Desired: 1
  • Balanced distribution across availability zones
  • Replaces failed instances automatically
Compute Layer (EC2)
  • Instance type: t2.micro
  • Web server: Apache installed via user-data script
  • Deployed across multiple availability zones
  • Security group allows traffic only from the load balancer

User-data scripts automatically configure each instance at launch, ensuring consistent environments.

Scaling Strategy
  • CPU-based scaling policy
  • Target CPU utilization: 50%
  • Instances launch automatically when load increases
  • Prevents over-provisioning while maintaining performance
Security Design
  • Internet traffic enters only through the ALB
  • EC2 instances accept inbound traffic only from ALB security group
  • No direct public access to backend instances
